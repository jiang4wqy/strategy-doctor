> 状态说明：这是最初排期记录。当前完成状态和发布步骤见 `docs/TEAM.md` 与 `handoff.md`。

> 一句话定位：**别人造策略，我们造给策略做死亡测试的引擎**——基于 Bitget 五大分析师 Skill 自动进化出最毒的市场剧情，告诉你任何策略会在什么时候、为什么死，并按你的风格开处方。

## 0. 产品闭环（所有排期都为这条线服务）

```
策略 ──▶ 红队/对抗层 ──▶ 回测/执行层 ──▶ 评分层 ──▶ 进化/处方 ──▶ held-out 复测
        (5个Skill=5攻击维度)  (复用Bitget        (3风格:稳健/      (定向修补)    (诚实取舍输出)
         自动进化毒剧情)       Agent Hub)         激进/趋势)
```

**MVP 验收线（必须能演的最短闭环）**：一个真实策略 → 自动诊断出死因 → 3 风格评分 → 单点处方 → 在**没参与治疗的** held-out 场景上复测，输出"稳健性↑X% / 收益−Y%"的诚实取舍。

## 1. 模块拆分（6 个功能块）

| # | 模块 | 干什么 | 关键产出 |
|---|---|---|---|
| A | **接口契约** | 冻结三大 schema + 仓库脚手架 | 策略 I/O、攻击场景、评分卡的类型文件 |
| B | **回测/执行层** | 包一层 Bitget Agent Hub，给(策略+场景)→标准 metrics | `backtest(strategy, scenario) → Metrics` |
| C | **红队/对抗层**（核心壁垒） | 5 个 Skill = 5 攻击维度，自动进化"毒剧情" | 符合场景 schema 的对抗剧情 + 自然语言死因 |
| D | **进化/处方** | 由红队死因驱动的定向修补（改阈值/加止损/降杠杆） | 处方 + held-out 复测结果 |
| E | **评分层** | 3 风格（稳健/激进/趋势）权重+阈值，给风险分 | 三风格评分卡 + 诚实取舍 |
| F | **产品/Demo** | 极简演示 + 3 分钟脚本 + 架构图 + 提交材料 | demo 录屏、pitch、docx 更新 |

## 3. 接口约定（"基因/攻击维度统一 I/O"——并行不打架的关键）

> 这是项目反复强调的承重点。已在 `src/contracts.ts` 冻结为 TS interface（v0.1），冻结后改动需全队同意。

### 3.1 策略契约（被测 + 被进化的对象）
```ts
interface Strategy {
  id: string; name: string;
  params: {                 // ←可调"基因"，进化引擎只动这里
    entry: object; exit: object; sizing: object;
    leverage: number; stopLoss: number; /* … */
  };
  universe: string[];       // 交易标的
  timeframe: string;
}
```

### 3.2 攻击场景契约（红队产出 → 回测消费）
```ts
interface Scenario {
  id: string; name: string;
  dimension: 'sentiment'|'macro'|'market-intel'|'news'|'technical';
  narrative: string;        // 自然语言死因剧情（LLM 生成，用于"解释"）
  marketShock: {            // 可回放的市场扰动参数（用于"回测"）
    type: string; magnitude: number; duration: string; path?: number[];
  };
  sourceSkill: string; severity: number;
}
```

### 3.3 评分卡契约（评分层产出）
```ts
interface Scorecard {
  strategyId: string; scenarioSetId: string;
  perStyle: {               // 同一策略，三风格不同权重→不同分
    conservative: StyleScore; aggressive: StyleScore; trend: StyleScore;
  };
  deaths: { scenarioId: string; cause: string }[];   // 死因列表
  prescription?: { changes: object[]; rationale: string; validatedOn: string };
  tradeoff: { robustnessGain: number; returnCost: number };  // 诚实取舍，禁止省略
}
interface StyleScore { riskScore: number; survived: boolean; maxDrawdown: number; }
```

### 3.4 回测适配器接口（B 模块对外唯一入口）
```ts
backtest(strategy: Strategy, scenario: Scenario): Metrics
// Metrics { pnl, maxDrawdown, sharpe, liquidated: boolean, … }
```

## 6. 风险与兜底

| 风险 | 兜底 |
|---|---|
| Bitget Agent Hub 接不通/限额 | 先 mock backtest，真实接入作增强；**Day 2 就验证**，别拖到后期 |
| LLM 红队幻觉/不稳定 | 先手写场景模板兜底，LLM 只做参数化+死因解释层 |
| 时间不够 | MVP 砍到"单策略+单风格+3 个手写场景+1 个处方"，保证能演 |
| 现场演示翻车 | 预录结果做 fallback，断网可放 |
| 自我过拟合（吹一键变好） | 治疗剧情≠验证剧情(held-out 分离)；**永远输出收益代价**——这是专业度加分点 |

## 7. 不可触碰的底线（项目红线）

- **严禁实盘真钱**：只读 API Key + 子账号，全程不下真实订单。
- **诚实优先**：治疗场景与验证场景必须分离；输出永远带"稳健性↔收益"取舍，不做"一键变好"话术。
- **卖铲子站位**：与 Bitget Playbook 互补非竞争，深用满 5 个 Skill = 平台加分。
